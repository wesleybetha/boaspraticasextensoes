Bem-vindo ao Guia de Boas Práticas!

Este guia interativo foi criado para ser uma fonte de consulta rápida e eficiente sobre os padrões de desenvolvimento e as melhores práticas adotadas pela nossa equipe.

Use a barra de pesquisa acima para encontrar qualquer termo no guia ou navegue pelos tópicos no menu lateral para encontrar as diretrizes que você precisa.

O objetivo é simples: escrever um código mais limpo, consistente e de fácil manutenção para todos.

1. Nomenclatura de Artefatos no Ambiente

Para manter a organização e rastreabilidade, a criação de relatórios e fontes de dados deve seguir um padrão estrito, associando o artefato ao ID da demanda.

Padrão de Nomenclatura

O nome do artefato deve ser prefixado com o identificador da demanda entre colchetes.

[ID-da-Demanda] Nome do Artefato

Exemplos

  • Relatório: [DE-12345] Relatório de Bens Patrimoniais
  • Fonte Dinâmica: [DE-12345] Fonte Relatório de Bens Patrimoniais

2. Variáveis e Parâmetros

A padronização na nomenclatura é fundamental para a legibilidade e manutenção do código. A convenção adotada deve ser consistente em todo o projeto.

Padrão camelCase

Nomes de variáveis e parâmetros devem seguir o padrão camelCase: iniciar com letra minúscula e capitalizar a primeira letra de cada palavra subsequente.

  • nomeUsuario
  • dataInicial
  • valorTotal
  • entidadeSelecionada

Recomendado: Clareza e Contexto

Nomes devem autoexplicar seu propósito, descrevendo o contexto.

// Nomes que descrevem o contexto
def nomeBem
def placaBem
def saldoAnteriorBem

Não Recomendado: Nomes Genéricos

Nomes que não dão contexto e podem gerar confusão.

// Nomes sem contexto
def nome
def placa
def saldo

Identificação de Estruturas de Dados

Indique o tipo da estrutura (List, Map) no nome da variável para maior clareza.

listBens = []
mapBensPorPlaca = [:]

4. Linguagem e Performance

A escolha da linguagem e a atenção às práticas de codificação impactam diretamente a performance, o consumo de memória e o tempo de processamento.

Linguagem de Programação: Groovy/Java

Para lógicas e customizações, a linguagem a ser utilizada deve ser Groovy/Java.

  • Vantagens: Maior flexibilidade, poder de processamento, e facilidade na manutenção e depuração de lógicas complexas.
  • Evitar: BFC-Script, devido a limitações de complexidade, performance e manutenibilidade a longo prazo.

Mapeamento Seletivo de Campos

Ao definir uma fonte de dados, mapeie apenas os campos que serão efetivamente utilizados no script para evitar tráfego de dados e consumo de memória desnecessários.

Evitar Consultas Aninhadas em Loops (N+1)

Nunca realize consultas dentro de um laço de repetição. Isso gera um número excessivo de queries ao banco de dados, causando lentidão severa.

Prática Ruim (N+1)

List bens = queryBens()
for(bem in bens){
  // Uma nova consulta para cada item!
  def infoExtra = queryInfoExtra(bem.id) 
  // ...
}

Alternativa Correta

// Carrega dados secundários antes
Map infosExtras = queryTodasInfosExtras()

List bens = queryBens()
for(bem in bens){
  // Acesso rápido em memória
  def infoExtra = infosExtras.get(bem.id)
  // ...
}

5. Comentários no Código

A regra de ouro é: comentários devem explicar o "porquê", não o "o quê". O código deve ser legível o suficiente para explicar o que está fazendo.

Comentário Bom

Esclarece a intenção por trás de uma regra de negócio complexa ou otimização.

// Itera sobre os bens para calcular o valor 
// depreciado, conforme a regra fiscal XYZ.

Comentário Ruim

Comentários óbvios que repetem o que o código faz não agregam valor.

// Itera sobre a lista de bens
// (O comando 'for' já diz isso).

i = i + 1; // Incrementa i

6. Parâmetros de Emissão de Relatórios

A definição dos parâmetros afeta a usabilidade, a performance e a manutenibilidade.

Nomenclatura e Padrão

Todos os parâmetros devem seguir o padrão camelCase.

Otimização e Usabilidade

Para garantir a performance, avalie sempre a possibilidade de tornar os parâmetros-chave obrigatórios (como período, entidade, status). Isso reduz a quantidade de dados retornados e resulta em relatórios mais rápidos e eficientes.

Consistência entre Artefatos

Os parâmetros criados no relatório (.jrxml) e na sua respectiva fonte dinâmica devem ser idênticos em nome (camelCase), tipo de dado e descrição para evitar erros e dificuldades de manutenção.

7. Padrão de Layout de Relatórios (Jaspersoft TIBCO)

Para garantir consistência visual e correta geração de PDFs, todo relatório deve ser baseado nos modelos `NossoPadraoInterno.jrxml` e `NossoPadraoExterno.jrxml`.

Modelos para Download

NossoPadraoInterno.jrxml NossoPadraoExterno.jrxml

Regras Gerais

  • Fonte Padrão: Utilizar a fonte Arial em todos os componentes.
  • Altura do TextField: A altura (`height`) deve ser sempre 3 pixels maior que o tamanho da fonte. (Ex: Fonte 9, Altura 12).
  • Alinhamento: Garantir alinhamento perfeito entre colunas do cabeçalho e dados.
  • Uso de Linhas (Line): Evitar na banda de detalhe para não degradar a performance.
  • Uso de Bordas (Border): Restrito ao cabeçalho. O corpo do relatório deve ser limpo.

Cabeçalho Externo (Visualização Final)

ESTADO: Fonte 14

ENTIDADE: Fonte 12

Nome do relatório: Fonte 10

Endereço, CEP, CNPJ, Telefone, E-mail, Site, Página, Data: Fonte 9

Parâmetros: Fonte 7, Altura 10px, Ajuste de Texto: StretchHeight

Cabeçalho Interno (Estrutura no .jrxml)

BTH_ENTIDADE_ESTADO: Fonte 14

ENTIDADE_NOME: Fonte 12

RELATÓRIO_TITULO: Fonte 10

Parâmetros: Fonte 7, Altura 10px, Ajuste de Texto: StretchHeight

Espaçamento Vertical: Manter 1 pixel entre os campos.

Dados, Totalizadores e Rodapé

  • Dados do Relatório: Fonte 7 a 9. Valor numérico à direita, texto à esquerda (ou centralizado se tiver máscara).
  • Valores Nulos: Numérico retorna `0,00`. Quantidade ou Data retornam em branco.
  • Totalizadores: Mesmas regras dos dados, mas em negrito.
  • Rodapé: Fonte 6. Alinhado à direita na página.
  • Seção "NoData": Renderizar a banda "Cabeçalho de Uso Interno".

9. Rotina de Publicação

O fluxo a seguir detalha os passos e responsabilidades para a publicação de um novo artefato, desde o desenvolvimento até a entrega final para todos os clientes.

  1. Conferir parâmetros da extensão DEV

    Verificar se foram realizados os vínculos entre relatório e fonte dinâmica, se tipagem e nomes das variáveis estão fechando.

  2. Subir para o GitLab DEV

    Publicar no GitLab o código da fonte dinâmica (.groovy) e o arquivo .jrxml do layout, seguindo os passos abaixo:

    1. Crie uma nova branch a partir da master do projeto do sistema desejado presente no subgrupo Orquestrador, dentro do projeto Contratos.
    2. Caso já possuir o projeto local, realizar o Git Pull para atualizar o local com base no remoto, antes de criar a branch.
    3. O nome da branch deve seguir o padrão: TipoBranch-CódigoTarefa. Ex: hotfix-COM-12345 para Incidentes e release-COM-12345 para Melhorias.
    4. Realize as alterações e faça o envio (commit) na nova branch.
    5. Envie (push) a branch para o repositório remoto.
    6. Abra um Merge Request (MR) para a master, certificando-se de vincular a demanda no título ou descrição. Deve seguir o padrão: #:CódigoTarefa: Detalhes ajuste. Ex: #:COM-12345: Resumo ajuste efetuado

  3. Publicar versão na Diretoria de Produtos DEV

    A partir da versão criada, deve ser gerada a versão oficial na base da entidade BETHA SISTEMAS - DIRETORIA DE PRODUTOS, garantindo o versão na entidade de controle de extensões.

    1. Conferir se existem atualizações da extensão na Diretoria de Produtos no mês da Sprint e se a extensão existe em mais de um sistema.
      • SIM: Revisar ambos os códigos e fazer o merge.
      • NÃO: Fazer a atualização normalmente replicando o que foi feito na homologação.
    2. Conferir parâmetros (Vínculos, ordenação, assinatura como última opção).

  4. Orquestrar para base da Prefeitura Municipal de Editais DEV

    Em primeiro momento, deve ser orquestrado apenas para a entidade citada, afim de efetuar validações.

  5. Realizar os testes em produção na base da Prefeitura Municipal de Demonstração de Editais DEV

    Com a publicação do artefato nesta entidade, devem ser efetuados as validações antes de publicar para todos os clientes.

  6. Publicar para todos os clientesDEV

    Após validações do artefato na Prefeitura Municipal de Demonstração de Editais, deve ser publicado para todos os clientes.

  7. Validar em produção na base de alguns clientesTester

    Executar o relatório publicado em alguns clientes, validando assim a publicação oficial em produção.

  8. Concluir as tarefas da sprint no JIRA DEV

    Ao concluir a tarefa, ela irá assumir o status Atendida, garantindo que a sprint encerre com sucesso.

10. Exceções às Regras

Este documento é um guia fundamental. No entanto, podem existir situações específicas onde uma regra, especialmente de layout, pode não resultar na melhor experiência para o cliente final.

Nesses casos raros, exceções podem ser consideradas, mas qualquer desvio das práticas estabelecidas deve ser previamente debatido e acordado entre todos os membros da equipe.
